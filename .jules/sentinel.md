## 2024-05-24 - System.CommandLine Response File Injection and Option Injection Bypass
**Vulnerability:** Argument injection and filter bypass were possible due to missing validation on input parameters passed to `ilspycmd`. In `RunRawCmd`, an attacker could pass an argument starting with `@` (e.g., `@response.rsp`) which `System.CommandLine` expands as a response file, allowing them to bypass the `dangerousArgs` filter and execute arbitrary restricted flags (like `-o` for arbitrary file write). Additionally, in `OpenAssembly`, `GetOrOpen`, and `ResolvePath`, the `assembly_path` argument wasn't checked for leading `-` or `@` characters, allowing attackers to inject options or response files when the `assembly_path` was appended to `ilspycmd` arguments.
**Learning:** Even when explicit dangerous arguments are filtered, underlying CLI libraries (like `System.CommandLine` used by `ilspycmd`) may have built-in features (like `@` response files) that can expand inputs into flags *after* the custom filter logic has run. Furthermore, positional arguments (like file paths) can be interpreted as flags if they start with a `-` or `@`.
**Prevention:** Always validate and sanitize all user input that will be passed to a subprocess command line, even if it is expected to be a positional argument. Block response file expansion (`@`) if not strictly necessary, and enforce that positional path arguments cannot begin with `-` or `@`. Use explicit positional delimiters like `--` if the target command line tool supports it.
